What kind of UI rework are we doing?

The game has 18 tabs, a dark green design system already in place, and a complex set of gameplay screens. What's the goal?

A

Apply the existing design system

The tokens (colors, spacing, fonts, radii) are already in constants/theme.ts. The spec says what to do to every screen. We implement it: SubTabBar, GameRow, ActionButton, StatCard, new HUD, remove ScreenHeader. All 18 screens get consistent treatment. No new features or layouts invented.

Pros

  • Spec already exists — just execute
  • Predictable scope
  • Immediate visual consistency

Cons

  • Won't fix deeper UX problems (18 tabs is a lot)
  • Navigation paradigm unchanged
B

Design system + navigation rethink

Apply the design system AND rethink the 18-tab structure. 18 tabs is too many for a bottom tab bar. Design a new navigation architecture — maybe 5–6 primary tabs with sub-navigation, or a dashboard home with jump points. Then implement the consistent visual treatment within that new structure.

Pros

  • Solves the real UX problem (tab overload)
  • More polished, app-like feel
  • Gives the game a coherent information architecture

Cons

  • More design work upfront
  • Navigation changes touch many files
C

Full visual redesign from scratch

Throw away the existing patterns. New visual language — different color palette, new card styles, new layouts per screen, possibly new animation system. Treat it as a v2 art direction pass, not just token application. The design system spec becomes a starting point, not the destination.

Pros

  • Maximum creative freedom
  • Can fix everything at once

Cons

  • Very large scope — months of work
  • Risk of over-designing
  • Needs very clear new visual direction first
