Implementation Approaches

Two systems, two decisions. Pick the approach for each before we go into full design detail.

☁️ System 1 — Climate Depth

How should temperature + multi-day streaks be modelled?

A

Temperature layer on top of existing events

Each WeatherDay gets minTemp and maxTemp alongside the existing event type. Frost events trigger when minTemp < crop.frostKillTemp. Drought accumulates as a soil moisture deficit over consecutive dry days. Events can streak 1–7 days. Forecast shows probability % that degrades with distance.

Pros

  • Fits cleanly onto existing data model
  • Temperature is the right realistic foundation
  • Sets up growing degree days later

Cons

  • Requires updating crop data (frostKillTemp per crop)

⭐ Recommended

B

Full meteorological model

Track temperature, humidity, rainfall mm, and wind speed separately. Weather event type is derived from these values — frost when minTemp < 0 + low humidity, drought when 10+ consecutive days <2mm rain. Highly realistic, weather feels emergent rather than categorical.

Pros

  • Most realistic possible
  • Emergent weather patterns

Cons

  • Large data model change
  • UI needs redesigning for new metrics
  • High implementation risk
C

Event streaks only (no temperature)

Keep the existing event types but let them persist for 2–7 days. A frost event lasts 3 days = 3× damage accumulation. No new data dimensions — lowest complexity.

Pros

  • Minimal code change
  • Easy to balance

Cons

  • Less realistic — no temperature nuance
  • Doesn't set up future features

📋 System 2 — Recurring Contracts

How should the contract + relationship system be structured in the codebase?

A

Separate recurring contract type

A new RecurringContract lives alongside existing one-off contracts. Buyers are persistent entities with a relationshipTier (New → Regular → Preferred → Exclusive). Each tier unlocks better bonuses, larger orders, disaster grace periods. Clean separation — one-off and recurring never interfere.

Pros

  • Clean separation of concerns
  • Buyers feel like real relationships
  • Easy to extend with new buyer types later

Cons

  • New data type to add to store

⭐ Recommended

B

Extend existing contracts with recurring flag

Add recurring: true, frequencyDays, and streakCount to the existing contract type. Reuses the same UI and store logic with minimal additions.

Pros

  • Less new code
  • Fits existing UI

Cons

  • Buyer relationships become awkward to track on contracts
  • Harder to show relationship tier clearly
